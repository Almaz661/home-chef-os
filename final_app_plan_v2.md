# ФИНАЛЬНЫЙ КОМПЛЕКСНЫЙ ПЛАН ПРИЛОЖЕНИЯ: Система управления домашней кухней

## 1. КОНЦЕПЦИЯ И ПОЗИЦИОНИРОВАНИЕ

Данный документ представляет собой всеобъемлющее техническое задание для веб-приложения, предназначенного для эффективного управления домашней кухней. Приложение ориентировано на русскоязычную семью, проживающую в Нидерландах, и призвано упростить процессы планирования меню, управления запасами продуктов, составления списков покупок и приготовления пищи. Основной платформой для использования является iPad, с дополнительной поддержкой Chromebook и оптимизацией для смартфонов (в частности, для списка покупок в магазине).

### 1.1. Отличия от конкурентов и уникальные преимущества

Наше приложение выделяется на фоне существующих решений благодаря глубокой интеграции ключевых функций и ориентации на специфические потребности целевой аудитории. Основные уникальные преимущества включают:

*   **Комплексное управление инвентарем с тремя хранилищами:** В отличие от большинства конкурентов, предлагающих лишь базовый учет продуктов, наше приложение разделяет инвентарь на три логические зоны: Холодильник, Морозилка и Кладовая. Это позволяет более точно отслеживать наличие продуктов и их сроки годности, а также интегрировать эти данные в процесс планирования меню [1].
*   **Локализованный сканер чеков с поддержкой голландского языка и конвертацией EUR/RUB:** Уникальная функция, разработанная специально для русскоязычной аудитории в Нидерландах. Сканер чеков с использованием OCR и LLM (GPT-4.1-mini) способен распознавать сокращения и названия продуктов на голландском языке, переводить их и сопоставлять с базой продуктов, а также автоматически конвертировать стоимость покупок из EUR в RUB с динамическим курсом [2] [3].
*   **Глубокая связка инвентаря и планирования меню:** Приложение не просто предлагает рецепты, но и активно использует данные об имеющихся продуктах в инвентаре для формирования предложений по меню, особенно для продуктов с истекающим сроком годности. Это помогает минимизировать пищевые отходы и оптимизировать бюджет [1] [2].
*   **Продвинутый импорт рецептов по URL:** Система импорта рецептов не ограничивается простым копированием, а использует микроразметку Schema.org (Recipe) и веб-скрапинг для извлечения полных данных (ингредиенты, шаги, фотографии) с любых сайтов, очищая их от рекламы и лишнего контента [4].

### 1.2. Целевая аудитория и сценарии использования

**Целевая аудитория:** Русскоязычные семьи, проживающие в Нидерландах, активно использующие iPad в быту, заинтересованные в оптимизации домашних процессов, экономии времени и бюджета, а также в минимизации пищевых отходов. Пользователи ценят удобство, автоматизацию и надежность.

**Сценарии использования:**

1.  **Еженедельное планирование меню:** Пользователь на iPad открывает приложение, просматривает базу рецептов (5000+), выбирает блюда на неделю, учитывая продукты в инвентаре и предложения по использованию остатков. Система автоматически формирует список покупок.
2.  **Поход в магазин:** Один из членов семьи берет смартфон, открывает приложение и использует оптимизированный список покупок. Благодаря оффлайн-режиму и семейной синхронизации, список всегда актуален, даже без стабильного интернет-соединения. После покупки сканирует чек, и продукты автоматически добавляются в инвентарь.
3.  **Приготовление пищи:** Пользователь ставит iPad на кухне, открывает режим готовки (Cooking Mode). Крупный текст, пошаговые инструкции, встроенные таймеры и голосовое управление позволяют готовить, не отвлекаясь на экран и не пачкая его руками.
4.  **Управление запасами:** Регулярная проверка инвентаря, отслеживание сроков годности, добавление новых продуктов вручную или через сканер чеков. Получение уведомлений о продуктах, которые скоро испортятся.
5.  **Поиск новых рецептов:** Пользователь находит интересный рецепт в интернете, копирует ссылку в приложение, и система автоматически импортирует его со всеми деталями.

## 2. ПРИНЦИПЫ ДИЗАЙНА (на основе ошибок конкурентов)

Разработка приложения будет основываться на принципах, призванных избежать распространенных ошибок, выявленных в ходе анализа конкурентов [2].

### 2.1. Что мы НЕ делаем (антипаттерны)

*   **НЕ блокируем экспорт данных:** Пользователи должны иметь полный контроль над своими данными и возможность экспортировать все рецепты, инвентарь и планы в стандартных форматах (JSON/ZIP) [2].
*   **НЕ делаем список покупок жестким:** Пользователь должен иметь возможность настраивать категории и сортировку списка покупок под свои предпочтения и особенности конкретных супермаркетов (например, Albert Heijn или Jumbo) [2].
*   **НЕ заставляем экран гаснуть во время готовки:** В режиме готовки экран iPad должен оставаться активным, чтобы избежать постоянных касаний грязными руками [2].
*   **НЕ игнорируем разные единицы измерения:** Система должна корректно обрабатывать и конвертировать различные единицы измерения (граммы, миллилитры, штуки, чашки и т.д.) для точного учета и объединения ингредиентов [2].
*   **НЕ создаем "закрытую экосистему":** Приложение должно быть гибким, позволяя пользователям добавлять рецепты вручную, импортировать их по URL без ограничений и интегрироваться с внешними сервисами, где это уместно [2].
*   **НЕ усложняем интерфейс на телефоне:** Мобильная версия приложения (для смартфона) должна быть максимально простой, быстрой и ориентированной на ключевые задачи (просмотр списка покупок) с крупными элементами управления [2].
*   **НЕ используем жесткие ограничения на масштабирование порций:** Масштабирование порций должно быть гибким, позволяя указывать любое количество порций, с учетом "кулинарной логики" для специй и мелких ингредиентов [2].
*   **НЕ допускаем неточного распознавания чеков:** Использование LLM для парсинга чеков должно обеспечить высокую точность распознавания голландских продуктов и их сопоставления с базой, минимизируя ручные исправления [3].

### 2.2. Принципы UX для iPad

Интерфейс для iPad будет разработан с учетом особенностей устройства и сценариев использования на кухне:

*   **Крупные кнопки и элементы управления:** Для удобства взаимодействия пальцами, особенно во время готовки.
*   **Минимум кликов:** Оптимизация пользовательских потоков для выполнения основных задач с минимальным количеством действий.
*   **Оффлайн-first подход:** Ключевые функции (просмотр рецептов, список покупок) должны быть доступны и полностью функциональны даже без интернет-соединения, благодаря PWA и Service Worker [1].
*   **Визуальная чистота и читаемость:** Использование Tailwind CSS 4 для создания современного, адаптивного и легко читаемого интерфейса с акцентом на контент.

### 2.3. Принципы работы с данными

*   **Экспорт данных:** Пользователи всегда могут экспортировать свои данные (рецепты, инвентарь, планы) в открытых форматах (JSON/ZIP) [2].
*   **Надежность и целостность:** Использование Drizzle ORM и MySQL обеспечивает надежное хранение данных и целостность связей между таблицами. Резервное копирование данных будет предусмотрено [1].
*   **Прозрачность:** Пользователь всегда понимает, как данные используются, откуда берутся (например, курс валют из API) и как их можно изменить. История изменений (где применимо) будет доступна.


## 3. АРХИТЕКТУРА ПРИЛОЖЕНИЯ

Приложение построено на современном технологическом стеке, обеспечивающем высокую производительность, масштабируемость и удобство разработки с фокусом на оптимизацию затрат.

**Технологический стек:** React 19 (фронтенд), Tailwind CSS 4 (стили), tRPC 11 (бэкенд API), Drizzle ORM (работа с БД), **SQLite + Litestream** (база данных). Хостинг на платформе Manus (или Hetzner VPS), кэширование и CDN через **Cloudflare**, хранение изображений в **Cloudflare R2** [1].

### 3.1. Модули и их связи (схема потоков данных)

Приложение состоит из следующих основных модулей, тесно интегрированных между собой:

*   **Dashboard:** Центральная панель, агрегирующая информацию из других модулей (предстоящие блюда, продукты с истекающим сроком годности, статистика).
*   **Рецепты:** Управление базой рецептов, импорт, поиск, фильтрация, масштабирование порций. Использование SSG для статической генерации страниц рецептов.
*   **Планирование меню:** Создание еженедельных планов, интеграция с инвентарем, генерация списка покупок.
*   **Режим готовки (Cooking Mode):** Пошаговые инструкции, таймеры, голосовое управление.
*   **Список покупок:** Автоматическое формирование, объединение ингредиентов, синхронизация.
*   **Сканер чеков:** Распознавание чеков с использованием Tesseract (основной) и Gemini-2.5-flash (fallback), добавление продуктов в инвентарь.
*   **Инвентарь:** Учет продуктов в 3 хранилищах, сроки годности, уровни запасов.
*   **Справочник продуктов:** Единая база продуктов, алиасы, единицы измерения, конвертация.
*   **Уведомления:** Система напоминаний (сроки годности, разморозка).
*   **Настройки и экспорт данных:** Управление пользовательскими параметрами, резервное копирование.

**Обновлённая схема потоков данных (с учётом оптимизации):**

```
Пользователь (iPad / Телефон / Chromebook)
        |
        v
[Cloudflare CDN / Edge Cache]
  - Статика: JS, CSS, иконки
  - SSG-страницы рецептов (HTML)
  - Изображения из R2
        |
        v
[Node.js / tRPC Server]  <-- Brotli-сжатие
  - Динамические API-запросы
  - Аутентификация
        |
        v
[SQLite + Litestream]     [Cloudflare R2]
  - Основная БД           - Изображения (WebP, 3 размера)
  - Кэш курса EUR/RUB     - Litestream backup
        |
        v
[Sharp Image Pipeline]    [Tesseract OCR]
  - 200px / 600px / 1200px  - Парсинг чеков (основной)
  - WebP конвертация         - Gemini-2.5-flash (fallback)
```

### 3.2. Структура БД (Drizzle ORM с SQLite)

База данных SQLite используется с Drizzle ORM для определения схемы и взаимодействия. Litestream обеспечивает непрерывное резервное копирование в Cloudflare R2.

```typescript

import { relations } from 'drizzle-orm';

// Пользователи
export const users = sqliteTable('users', {
  id: integer('id').primaryKey({ autoIncrement: true }),

  cookTimeMinutes: integer('cook_time_minutes'),
  servings: integer('servings'),
  cuisineType: text('cuisine_type'),
  dietaryRestrictions: text('dietary_restrictions', { mode: 'json' }),
  caloriesPerServing: integer('calories_per_serving'),
  createdAt: integer('created_at', { mode: 'timestamp' }).default(new Date()),
  updatedAt: integer('updated_at', { mode: 'timestamp' }),
});

// Ингредиенты рецепта
export const recipeIngredients = sqliteTable('recipe_ingredients', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  recipeId: integer('recipe_id').notNull().references(() => recipes.id, { onDelete: 'cascade' }),
  productMasterId: integer('product_master_id').references(() => productMaster.id),
  rawName: text('raw_name').notNull(),
  amount: text('amount'),
  unit: text('unit'),
  notes: text('notes'),
});

// Шаги приготовления
export const recipeSteps = sqliteTable('recipe_steps', {

  stepNumber: integer('step_number').notNull(),

// Справочник продуктов
export const productMaster = sqliteTable('product_master', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  nameRu: text('name_ru').notNull().unique(),
  nameNl: text('name_nl'),
  nameEn: text('name_en'),
  defaultUnit: text('default_unit'),
  category: text('category'),
});

// Алиасы продуктов (для сканера чеков)
export const productAliases = sqliteTable('product_aliases', {

// Инвентарь (3 хранилища)
export const inventory = sqliteTable('inventory', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  userId: integer('user_id').notNull().references(() => users.id, { onDelete: 'cascade' }),
  productMasterId: integer('product_master_id').notNull().references(() => productMaster.id),
  quantity: integer('quantity').notNull(),
  unit: text('unit').notNull(),
  storageLocation: text('storage_location').notNull(), // 'fridge' | 'freezer' | 'pantry'
  expirationDate: integer('expiration_date', { mode: 'timestamp' }),

// Список покупок
export const shoppingListItems = sqliteTable('shopping_list_items', {
  id: integer('id').primaryKey({ autoIncrement: true }),

// Планирование меню
export const mealPlans = sqliteTable('meal_plans', {

// История сканирования чеков
export const receiptScans = sqliteTable('receipt_scans', {
  id: integer('id').primaryKey({ autoIncrement: true }),
```

### 3.3. API-эндпоинты (tRPC роутеры)

Все взаимодействие между фронтендом и бэкендом осуществляется через tRPC, обеспечивая строгую типизацию и высокую производительность. Основные группы роутеров:

*   **`recipesRouter`:** `getById`, `getAll` (с фильтрами), `importFromUrl` (Schema.org + AI fallback), `updateServings`, `create`, `update`, `delete`.
*   **`mealPlanningRouter`:** `getPlan`, `addMeal`, `updateMeal`, `deleteMeal`, `suggestMeals` (на основе инвентаря).
*   **`shoppingListRouter`:** `getList`, `generateFromPlan`, `addItem`, `updateItem`, `deleteItem`, `mergeItems`.
*   **`inventoryRouter`:** `getInventory` (по хранилищу), `addItem`, `updateItem`, `deleteItem`, `getExpiringItems`.
*   **`receiptScannerRouter`:** `uploadReceipt`, `processReceipt` (Tesseract + Gemini fallback, объединённый промпт), `getScanResult`, `confirmReceipt`.
*   **`productMasterRouter`:** `search`, `getUnits`, `convertUnits`.
*   **`notificationsRouter`:** `getNotifications`, `markAsRead`.
*   **`settingsRouter`:** `updateUserSettings`, `exportData`.

### 3.4. PWA/Service Worker стратегия для оффлайна

Для обеспечения бесперебойной работы приложения, особенно списка покупок в магазине, реализована стратегия Progressive Web App (PWA) с использованием Service Worker [1]. PWA также устраняет необходимость в публикации в App Store и Google Play.

*   **Кэширование ресурсов:** Service Worker кэширует статические ресурсы (HTML, CSS, JavaScript, изображения) при первом посещении, обеспечивая мгновенную загрузку при последующих визитах.
*   **Оффлайн-доступ к данным:** Ключевые данные (база рецептов частично, текущий план меню, список покупок) кэшируются в IndexedDB. Пользователи могут просматривать и взаимодействовать с этими данными без интернет-соединения.
*   **Фоновая синхронизация:** При появлении интернет-соединения Service Worker автоматически синхронизирует изменения, сделанные в оффлайн-режиме (например, отмеченные пункты в списке покупок).
*   **Push-уведомления:** Отправка push-уведомлений для напоминаний (сроки годности, разморозка) даже когда приложение закрыто.

## 4. ДЕТАЛЬНОЕ ОПИСАНИЕ КАЖДОГО МОДУЛЯ

В данном разделе представлено подробное описание каждого модуля приложения, его функциональности, интерфейса, логики работы и связей с другими компонентами, а также подходы к избеганию ошибок, выявленных у конкурентов.

### 4.1. Dashboard (главная)

**Что делает:** Предоставляет пользователю быстрый обзор текущего состояния кухни и предстоящих задач. Является центральной точкой входа в основные функции приложения.

**Экраны/Интерфейс:**
*   **6 информационных карточек:** Отображают ключевую информацию, такую как: "Что готовим сегодня/завтра", "Продукты с истекающим сроком годности", "Предложения по использованию остатков", "Текущий список покупок", "Статистика расходов за неделю", "Напоминания".
*   **Быстрый доступ:** Каждая карточка является интерактивной и позволяет перейти к соответствующему модулю для детального просмотра или действия.
*   **Настраиваемость:** Пользователь может настраивать порядок и видимость карточек.

**Логика:** Агрегирует данные из модулей "Планирование меню", "Инвентарь", "Список покупок", "Сканер чеков" и "Уведомления". Обновляется в реальном времени при изменении данных в связанных модулях.

**Связи с другими модулями:**
*   **Планирование меню:** Отображает предстоящие блюда, позволяет быстро перейти к редактированию плана.
*   **Инвентарь:** Показывает продукты, требующие внимания (истекающий срок годности), предлагает рецепты из них.
*   **Список покупок:** Отображает количество позиций в списке, позволяет быстро перейти к просмотру.
*   **Сканер чеков:** Может отображать последнюю отсканированную сумму или напоминание о сканировании.
*   **Уведомления:** Отображает активные уведомления.

**Как избегаем ошибок конкурентов:** Предоставляем настраиваемый и информативный дашборд, который не перегружен лишней информацией и позволяет быстро перейти к действию, в отличие от некоторых конкурентов, где дашборд либо отсутствует, либо не несет практической ценности [2].

### 4.2. Рецепты (база 5000+, импорт по URL, поиск, фильтры, масштабирование порций)

**Что делает:** Централизованное хранилище всех рецептов пользователя, с возможностью добавления, поиска, фильтрации, редактирования и масштабирования.

**Экраны/Интерфейс:**
*   **Список рецептов:** Отображение рецептов в виде карточек с фотографиями, названием, временем готовки и рейтингом. Возможность переключения между режимами просмотра (сетка/список).
*   **Детальная страница рецепта:** Полное описание рецепта, включая ингредиенты (с возможностью добавления в список покупок), пошаговые инструкции (с таймерами и фото), информацию о пищевой ценности, источник. Кнопки для добавления в план меню, масштабирования порций, редактирования.
*   **Форма импорта по URL:** Поле для ввода URL, кнопка "Импортировать". Отображение превью импортированного рецепта для подтверждения.
*   **Форма создания/редактирования рецепта:** Поля для ввода названия, описания, ингредиентов, шагов, загрузки фото, указания времени готовки, порций, тегов.
*   **Поиск и фильтры:** Строка поиска по названию, ингредиентам. Расширенные фильтры по типу кухни, диетическим ограничениям, времени готовки, калорийности.

**Логика:**
*   **Импорт по URL:** Использует микроразметку Schema.org и веб-скрапинг для извлечения данных. Скачивает изображения в S3. Сохраняет данные в таблицы `recipes`, `recipeIngredients`, `recipeSteps` [4].
*   **Масштабирование порций:** При изменении количества порций автоматически пересчитывает количество ингредиентов. Учитывает "кулинарную логику" для специй и мелких ингредиентов (например, 0.3 яйца округляется до 1) [2].
*   **Поиск и фильтрация:** Оптимизированный поиск по тексту и фильтрация по множеству параметров, включая связь с `productMaster` для поиска по ингредиентам.

**Связи с другими модулями:**
*   **Планирование меню:** Рецепты добавляются в план.
*   **Список покупок:** Ингредиенты из выбранных рецептов добавляются в список.
*   **Режим готовки:** Детальная страница рецепта является основой для режима готовки.
*   **Справочник продуктов:** Ингредиенты рецептов связываются с `productMaster` для унификации и конвертации единиц измерения.

**Как избегаем ошибок конкурентов:**
*   **Надежный импорт:** В отличие от конкурентов с проблемным импортом [2], наш парсер будет использовать продвинутые методы (Schema.org + веб-скрапинг) и сохранять все данные, включая шаги и фото [4].
*   **Гибкое масштабирование:** Избегаем жестких ограничений на количество порций, предлагая гибкий пересчет с учетом кулинарных особенностей [2].
*   **Полноценное редактирование:** Пользователь может полностью редактировать импортированные или созданные рецепты, в отличие от систем, где это невозможно или затруднено [2].

### 4.3. Планирование меню (динамическая таблица, предложения из остатков)

**Что делает:** Позволяет пользователю создавать еженедельные планы питания, интегрируя рецепты с инвентарем и генерируя списки покупок.

**Экраны/Интерфейс:**
*   **Динамическая таблица планирования:** Календарная сетка (например, 7 дней на 3-4 приема пищи: Завтрак, Обед, Ужин, Перекус). Ячейки динамически расширяются для отображения нескольких блюд. Возможность горизонтального скролла для iPad [2].
*   **Drag-and-drop:** Перетаскивание рецептов из базы в ячейки плана.
*   **Предложения рецептов:** Боковая панель или модальное окно с предложениями рецептов на основе продуктов в инвентаре (особенно с истекающим сроком годности) [1].
*   **Кнопка "Сгенерировать список покупок":** Автоматически формирует список на основе выбранных рецептов.

**Логика:**
*   **Динамические слоты:** Таблица не имеет жестких ограничений на количество строк, позволяя добавлять любое количество приемов пищи или блюд в ячейку [1].
*   **Интеграция с инвентарем:** При добавлении рецепта в план, система проверяет наличие ингредиентов в инвентаре и подсвечивает недостающие. Предлагает рецепты, использующие продукты с истекающим сроком годности [1].
*   **"Умные подсказки" вместо автоматического вычитания:** При приготовлении блюда, система предложит подтвердить расход продуктов из инвентаря, а не вычтет их автоматически, чтобы избежать рассинхронизации с реальностью [3].

**Связи с другими модулями:**
*   **Рецепты:** Основной источник блюд для планирования.
*   **Инвентарь:** Использует данные инвентаря для предложений и проверки наличия ингредиентов.
*   **Список покупок:** Генерирует список на основе плана.
*   **Уведомления:** Может генерировать напоминания о необходимости составить план.

**Как избегаем ошибок конкурентов:**
*   **Гибкий интерфейс:** Избегаем перегруженности и жестких ограничений, характерных для некоторых конкурентов [2].
*   **Реалистичное управление инвентарем:** Отказываемся от автоматического вычитания в пользу подтверждения расхода, что решает проблему рассинхронизации, выявленную в анализе [3].
*   **Минимизация отходов:** Активно предлагаем рецепты из продуктов с истекающим сроком годности, что является ключевым преимуществом [1].

### 4.4. Режим готовки (Cooking Mode: пошаговый, таймеры, hands-free, блокировка экрана)

**Что делает:** Предоставляет оптимизированный интерфейс для пошагового выполнения рецепта, минимизируя необходимость взаимодействия с экраном во время готовки.

**Экраны/Интерфейс:**
*   **Полноэкранный режим:** Крупный, легко читаемый текст шагов. Фотографии шагов (если есть).
*   **Навигация:** Крупные кнопки "Предыдущий шаг" / "Следующий шаг". Возможность голосового управления для переключения шагов [1].
*   **Встроенные таймеры:** Автоматически распознает время в тексте рецепта и создает кликабельные таймеры, которые можно запустить прямо из шага [1].
*   **Блокировка экрана:** Предотвращает автоотключение экрана iPad во время активного режима готовки [1].

**Логика:**
*   **Парсинг шагов:** Использует данные из таблицы `recipeSteps` для отображения пошаговых инструкций.
*   **Голосовое управление:** Интеграция с API распознавания речи для команд типа "следующий шаг", "поставь таймер на 5 минут" [1].
*   **Таймеры:** Управление несколькими таймерами одновременно, отображение их статуса.

**Связи с другими модулями:**
*   **Рецепты:** Получает пошаговые инструкции и данные о времени готовки.
*   **Уведомления:** Может использовать для оповещения об окончании таймера.

**Как избегаем ошибок конкурентов:**
*   **Hands-free:** Внедряем голосовое управление и блокировку экрана, решая проблему необходимости касаться экрана грязными руками, на которую жаловались пользователи конкурентов [2].
*   **Интегрированные таймеры:** Предоставляем удобные таймеры прямо в контексте шагов, что является преимуществом по сравнению с базовыми решениями [1].

### 4.5. Список покупок (автозаполнение, объединение ингредиентов, оффлайн, семейная синхронизация)

**Что делает:** Автоматически генерирует и управляет списком продуктов, необходимых для покупки, с учетом имеющихся запасов и потребностей семьи.

**Экраны/Интерфейс:**
*   **Список продуктов:** Отображение продуктов, сгруппированных по категориям (например, "Молочные продукты", "Овощи", "Бакалея"). Возможность перетаскивания для изменения порядка или категории [2].
*   **Отметка купленного:** Чекбоксы для отметки купленных товаров. Автоматическое перемещение в раздел "Куплено" или скрытие.
*   **Добавление вручную:** Поле для быстрого добавления продуктов, с автодополнением из `productMaster`.
*   **Мобильная версия:** Упрощенный интерфейс для смартфона с крупными кнопками и высокой контрастностью для использования в магазине [2].

**Логика:**
*   **Автозаполнение:** Генерируется на основе выбранных рецептов в "Планировании меню" и недостающих продуктов из "Инвентаря" [1].
*   **Объединение ингредиентов:** Интеллектуальный алгоритм объединяет одинаковые продукты из разных рецептов, приводя их к единым единицам измерения (например, "2 яйца" + "3 яйца" = "5 яиц"). Пользователь может вручную корректировать объединение [2].
*   **Оффлайн-режим:** Благодаря PWA и Service Worker, список полностью функционален без интернета. Изменения синхронизируются при появлении сети [1].
*   **Семейная синхронизация:** Real-time синхронизация (WebSockets/tRPC subscriptions) позволяет нескольким членам семьи одновременно работать со списком, видеть изменения и отмечать купленные товары [1].

**Связи с другими модулями:**
*   **Планирование меню:** Основной источник для автозаполнения.
*   **Инвентарь:** Используется для определения недостающих продуктов.
*   **Справочник продуктов:** Для унификации названий, единиц измерения и категорий.
*   **Сканер чеков:** Добавляет купленные продукты в инвентарь после сканирования.

**Как избегаем ошибок конкурентов:**
*   **Умное объединение:** Решаем проблему дублирования и некорректного объединения ингредиентов, на которую жаловались пользователи Paprika [2].
*   **Гибкость:** Позволяем настраивать категории и сортировку, избегая жестких ограничений Cooklist [2].
*   **Оффлайн и синхронизация:** Обеспечиваем надежную работу в магазине и совместное использование, что является слабым местом многих приложений [2].

### 4.6. Сканер чеков (OCR + LLM, голландский, EUR/RUB, динамический курс)

**Что делает:** Позволяет быстро добавлять купленные продукты в инвентарь путем сканирования бумажных чеков, автоматически распознавая товары, цены и конвертируя валюту.

**Экраны/Интерфейс:**
*   **Интерфейс камеры:** Для фотографирования чека.
*   **Экран подтверждения:** Отображение распознанных позиций чека с возможностью ручного редактирования (название, количество, цена, категория). Подсветка нераспознанных или сомнительных позиций.
*   **Отображение курса:** Текущий курс EUR/RUB и курс, примененный к чеку.

**Логика:**
*   **OCR:** Использует технологию оптического распознавания символов для извлечения текста с изображения чека.
*   **LLM-парсинг:** Распознанный текст чека передается в LLM (например, GPT-4.1-mini) с промптом для расшифровки сокращений, перевода голландских названий на русский и структурирования данных в JSON (название, количество, цена) [3].
*   **Сопоставление с `productMaster`:** Распознанные продукты сопоставляются с `productMaster` и `productAliases` для унификации. Если продукт не найден, предлагается создать новый или выбрать из существующих.
*   **Динамический курс валют:** Интеграция с API (например, ExchangeRate-API) для получения актуального курса EUR/RUB. Курс на момент сканирования сохраняется в БД для исторических отчетов [3].
*   **Добавление в инвентарь:** После подтверждения, продукты автоматически добавляются в инвентарь пользователя.

**Связи с другими модулями:**
*   **Инвентарь:** Основной получатель данных о купленных продуктах.
*   **Справочник продуктов:** Используется для сопоставления и унификации названий продуктов.
*   **История сканирования чеков:** Сохраняет данные о сканированных чеках для анализа расходов.

**Как избегаем ошибок конкурентов:**
*   **Высокая точность распознавания:** Использование LLM значительно повышает точность распознавания голландских чеков и их перевода, решая проблему ненадежности OCR, на которую жаловались пользователи [3].
*   **Динамический курс:** Избегаем жестко закодированного курса валют, обеспечивая актуальность финансовых данных [3].
*   **Ручное подтверждение:** Предоставляем возможность ручной корректировки, чтобы пользователь мог исправить ошибки распознавания, что повышает доверие к системе [3].

### 4.7. Инвентарь (3 хранилища, сроки годности, уровни запасов, связь с планированием)

**Что делает:** Позволяет пользователю вести точный учет всех продуктов, имеющихся дома, с разделением по местам хранения и отслеживанием сроков годности.

**Экраны/Интерфейс:**
*   **Разделы хранилищ:** Отдельные вкладки или секции для Холодильника, Морозилки, Кладовой.
*   **Список продуктов:** Отображение продуктов с указанием количества, единицы измерения, срока годности. Цветовая индикация для продуктов с истекающим сроком годности.
*   **Добавление/редактирование продукта:** Форма для ввода названия, количества, единицы измерения, места хранения, срока годности. Автодополнение из `productMaster`.
*   **Быстрая ревизия:** Функция, предлагающая быстро пройтись по категориям и подтвердить наличие продуктов [3].

**Логика:**
*   **3 хранилища:** Четкое разделение продуктов по `storageLocation` в таблице `inventory`.
*   **Сроки годности:** Система отслеживает `expirationDate` и генерирует уведомления о приближающемся истечении срока годности [1].
*   **Уровни запасов:** Возможность задавать минимальные уровни запасов для часто используемых продуктов, чтобы система могла напоминать о необходимости пополнения.
*   **Учет готовых блюд:** Специальный раздел в "Морозилке" для учета замороженных готовых блюд, которые можно быстро интегрировать в план меню [1].

**Связи с другими модулями:**
*   **Сканер чеков:** Основной источник для пополнения инвентаря.
*   **Планирование меню:** Используется для проверки наличия ингредиентов и предложений рецептов.
*   **Список покупок:** Определяет, какие продукты необходимо купить.
*   **Уведомления:** Генерирует напоминания о сроках годности и необходимости пополнения.
*   **Справочник продуктов:** Для унификации названий и единиц измерения.

**Как избегаем ошибок конкурентов:**
*   **Точный учет:** В отличие от систем, где инвентарь быстро рассинхронизируется [3], мы предлагаем функцию "быстрой ревизии" и подтверждение расхода, чтобы поддерживать актуальность данных.
*   **Минимизация отходов:** Активно используем данные о сроках годности для предложений рецептов, что является важным преимуществом [1].
*   **Учет готовых блюд:** Расширяем функционал инвентаря для учета замороженных блюд, что отсутствует у многих конкурентов [1].

### 4.8. Справочник продуктов (единая база продуктов, алиасы, единицы измерения, конвертация)

**Что делает:** Единая централизованная база данных всех продуктов, используемых в приложении, обеспечивающая унификацию названий, единиц измерения и возможность конвертации.

**Экраны/Интерфейс:**
*   **Список продуктов:** Возможность просмотра и поиска по `productMaster`.
*   **Форма редактирования/добавления:** Поля для русского, голландского, английского названий, категории, единицы измерения по умолчанию. Возможность добавления алиасов.

**Логика:**
*   **Унификация:** Все модули (Рецепты, Инвентарь, Список покупок, Сканер чеков) ссылаются на `productMaster` для обеспечения единообразия данных.
*   **Алиасы:** Таблица `productAliases` позволяет сопоставлять различные варианты названий (например, сокращения из чеков, региональные названия) с основным продуктом [3].
*   **Конвертация единиц измерения:** Встроенная логика для конвертации между различными единицами измерения (например, граммы в штуки, миллилитры в чашки) на основе предопределенных коэффициентов или пользовательских настроек [2].

**Связи с другими модулями:**
*   **Рецепты:** Ингредиенты рецептов связываются с `productMaster`.
*   **Инвентарь:** Продукты в инвентаре ссылаются на `productMaster`.
*   **Список покупок:** Элементы списка покупок связываются с `productMaster`.
*   **Сканер чеков:** Использует `productMaster` и `productAliases` для распознавания и сопоставления продуктов.

**Как избегаем ошибок конкурентов:**
*   **Унификация и конвертация:** Решаем проблему несовместимости единиц измерения и дублирования продуктов, на которую жаловались пользователи [2].
*   **Поддержка алиасов:** Значительно улучшаем точность распознавания чеков и поиска, обрабатывая различные варианты названий [3].

### 4.9. Уведомления (сроки годности, напоминания, разморозка)

**Что делает:** Информирует пользователя о важных событиях и предстоящих задачах, помогая эффективно управлять кухней и минимизировать отходы.

**Экраны/Интерфейс:**
*   **Центр уведомлений:** Список активных уведомлений в приложении.
*   **Push-уведомления:** Всплывающие уведомления на устройстве (iPad, смартфон) даже когда приложение закрыто (через Service Worker).

**Логика:**
*   **Сроки годности:** Генерирует уведомления о продуктах в инвентаре, срок годности которых истекает в ближайшее время [1].
*   **Напоминания:** Пользователь может создавать произвольные напоминания (например, "разморозить мясо", "составить меню на неделю").
*   **Интеграция с планированием:** Может напоминать о необходимости составить план меню или о предстоящих блюдах.

**Связи с другими модулями:**
*   **Инвентарь:** Основной источник для уведомлений о сроках годности.
*   **Планирование меню:** Может генерировать напоминания о планировании.
*   **Режим готовки:** Может использоваться для таймеров.

**Как избегаем ошибок конкурентов:** Предоставляем гибкую и настраиваемую систему уведомлений, которая активно помогает пользователю, а не просто является пассивным списком [1].

## 7. ЧЕК-ЛИСТ КАЧЕСТВА

На основе анализа слабых сторон конкурентов и выявленных "антипаттернов" разработан чек-лист качества, который необходимо использовать перед каждым релизом для обеспечения высокого уровня пользовательского опыта и надежности приложения.

*   [ ] **Экспорт данных:** Проверена работоспособность функции экспорта всех пользовательских данных (рецепты, инвентарь, планы) в форматах JSON/ZIP. Данные легко читаемы и могут быть импортированы обратно или в другие системы.
*   [ ] **Гибкость списка покупок:** Проверена возможность пользователя настраивать категории и сортировку списка покупок. Убедиться, что список не привязан жестко к одной структуре магазина.
*   [ ] **Режим готовки (Cooking Mode):** Проверена блокировка автоотключения экрана iPad во время активного режима готовки. Проверена работоспособность голосового управления (если реализовано) и встроенных таймеров.
*   [ ] **Единицы измерения:** Проверена корректность обработки и конвертации различных единиц измерения во всех модулях (рецепты, инвентарь, список покупок). Отсутствие дублирования ингредиентов из-за разных единиц измерения.
*   [ ] **Импорт рецептов:** Проверена надежность импорта рецептов по URL с различных сайтов. Убедиться, что все данные (ингредиенты, шаги, фото) извлекаются корректно и без потерь. Отсутствие рекламы и лишнего текста.
*   [ ] **Мобильный интерфейс (смартфон):** Проверена простота и удобство использования списка покупок на смартфоне. Крупные элементы управления, высокая контрастность, минимум кликов.
*   [ ] **Масштабирование порций:** Проверена корректность пересчета ингредиентов при изменении количества порций, включая "кулинарную логику" для специй и мелких ингредиентов.
*   [ ] **Точность сканера чеков:** Проверена точность распознавания голландских названий продуктов и их сопоставления с `productMaster` и `productAliases`. Минимальное количество ручных исправлений, требуемых от пользователя. Корректность конвертации EUR/RUB с динамическим курсом.
*   [ ] **Синхронизация данных:** Проверена надежность оффлайн-режима и фоновой синхронизации данных (особенно для списка покупок и инвентаря). Проверена real-time семейная синхронизация (если реализована).
*   [ ] **Инвентарь:** Проверена актуальность данных в инвентаре после добавления продуктов через сканер чеков или ручного ввода. Проверена работоспособность уведомлений о сроках годности. Проверена функция "быстрой ревизии" (если реализована).
*   [ ] **Производительность:** Проверена скорость работы приложения при больших объемах данных (например, 5000+ рецептов, большой инвентарь). Отсутствие задержек при поиске и фильтрации.
*   [ ] **Безопасность:** Проверена защита пользовательских данных, включая хранение изображений в S3 и безопасность API-эндпоинтов.


## 8. ОПТИМИЗАЦИЯ ЗАТРАТ И ЭКОНОМИЯ

В целях минимизации операционных расходов (OPEX) и обеспечения финансовой устойчивости проекта, архитектура и выбор технологий были пересмотрены. Ниже представлены ключевые решения по оптимизации затрат.

### 8.1. Архитектурные решения для экономии

1.  **База данных: SQLite + Litestream.** Замена MySQL на встроенную базу данных SQLite устраняет необходимость в отдельном сервере базы данных, что значительно снижает ежемесячные расходы. Для обеспечения надежности и резервного копирования используется Litestream, который непрерывно реплицирует данные в S3-совместимое хранилище (Cloudflare R2). Drizzle ORM полностью поддерживает SQLite без изменений в коде приложения.

2.  **CDN и кэширование: Cloudflare (Free Tier).** Использование бесплатного тарифа Cloudflare для edge-кэширования статических ресурсов (JS, CSS, изображения). Это снижает нагрузку на основной сервер в 5-10 раз, позволяя использовать более дешевый тарифный план хостинга.

3.  **Генерация статических страниц (SSG).** Для базы из 5000+ рецептов применяется подход Static Site Generation (SSG). Страницы рецептов генерируются в статический HTML при импорте или обновлении и отдаются напрямую через CDN Cloudflare, минуя бэкенд. Это радикально снижает нагрузку на сервер БД и API.

4.  **Оптимизация изображений: Sharp Pipeline.** Внедрение библиотеки Sharp для обработки изображений на бэкенде. При загрузке оригинального изображения (например, из рецепта или чека) автоматически создаются три оптимизированные версии в формате WebP: миниатюра (200px), средний размер (600px) и полноразмерное (1200px). Это обеспечивает экономию места в хранилище в 5-8 раз и ускоряет загрузку на клиенте.

5.  **Хранение данных: Cloudflare R2.** Переход с традиционного AWS S3 на Cloudflare R2 для хранения изображений и бэкапов базы данных (Litestream). R2 предлагает бесплатный уровень до 10 ГБ и отсутствие платы за исходящий трафик (egress fees), что делает его идеальным выбором для проекта с большим количеством медиаконтента.

6.  **Парсинг чеков: Tesseract + LLM Fallback.** Использование бесплатного OCR-движка Tesseract в качестве основного инструмента для извлечения текста из чеков. LLM применяется только как fallback-механизм для сложных случаев (нечеткий текст, сложные сокращения). Это существенно сокращает количество платных API-запросов.

7.  **Оптимизация LLM-запросов.** При необходимости использования LLM для парсинга чеков применяется объединение задач: OCR-распознавание, перевод на русский и сопоставление с базой продуктов выполняются в рамках одного составного промпта, а не трёх отдельных запросов. Это снижает стоимость обработки одного чека в 3 раза.

8.  **Выбор модели LLM.** Использование модели Gemini-2.5-flash вместо более дорогой GPT-4.1-mini там, где это возможно без критической потери качества (например, для базовой классификации, перевода или парсинга чеков).

9.  **Импорт рецептов: Schema.org + AI Fallback.** Основной метод импорта рецептов — парсинг микроразметки Schema.org (бесплатно, покрывает ~80% сайтов). Использование AI для извлечения данных применяется только для оставшихся 20% сайтов без разметки.

10. **Кэширование курсов валют.** Для функции конвертации EUR/RUB курс запрашивается из API только 1 раз в сутки и кэшируется в базе данных (таблица `exchangeRates`). Это позволяет избежать превышения лимитов бесплатных API курсов валют.

11. **Сжатие трафика: Brotli.** Использование алгоритма сжатия Brotli вместо Gzip на уровне сервера и CDN. Это обеспечивает дополнительную экономию трафика на 15-20% при сопоставимой нагрузке на CPU.

12. **Мониторинг (Free Tier).** Использование бесплатных тарифов сервисов мониторинга: UptimeRobot для проверки доступности и Sentry (Free Tier) для отслеживания ошибок. Для текущего масштаба проекта этих инструментов более чем достаточно.

13. **Отказ от нативных приложений: PWA.** Фокус на развитии Progressive Web App (PWA) вместо разработки нативных приложений для iOS/Android. Это экономит $99/год на аккаунте Apple Developer и позволяет избежать 30% комиссии магазинов приложений, обеспечивая при этом полноценную оффлайн-функциональность и возможность установки на домашний экран.

14. **Хостинг.** В случае необходимости миграции с платформы Manus, использование доступных VPS-провайдеров, таких как Hetzner (локация в Нидерландах, от 3.79 EUR/мес), что значительно дешевле управляемых облачных решений.
